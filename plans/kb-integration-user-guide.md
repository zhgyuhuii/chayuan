# 察元 AI 知识库集成 — 使用指南

面向最终用户与 QA。完整设计请见 `plans/plan-knowledge-base-integration.md`。

---

## 1. 一分钟上手

1. **配置连接**：WPS → 察元 AI → 设置 → 常规设置 → 知识库设置 → 添加
2. **验证可达**：保存后点"测试连接"，三步全绿即可
3. **绑定知识库**：察元 AI 助手对话框 → 输入框右上的"知识库"按钮 → 勾选要用的 KB
4. **发送问题**：正常发消息，模型回答会带 `[c1]` 等可点击的引用脚标，对话气泡上方多出一条折叠条 — 展开看引用片段、信任度星级、原始附件下载

## 2. 设置详解

### 2.1 连接类型

| 类型 | 适用场景 | 必填字段 |
| --- | --- | --- |
| **JWT（用户）** | 个人账号登录 chayuan-server，使用自己有权访问的 KB | 服务器地址 / 用户名 / 密码 |
| **HMAC（应用）** | 部门级共享账号，由管理员预先发放 App ID + AppKey | 服务器地址 / App ID / App Secret |

> 凭据落盘前用 AES-GCM + HKDF 加密，密钥派生自设备绑定 OPFS 盐，**不会**以明文形式同步到任何位置。

### 2.2 连通性测试三步

| 步骤 | 检查内容 | 失败常见原因 |
| --- | --- | --- |
| 1. **服务地址** | `GET /healthz` 200 | URL 错 / 服务未启动 / 网络不可达 |
| 2. **凭据** | JWT login 或 HMAC ping | 用户名密码错 / App Secret 错 / 时钟漂移 > 5 分钟 |
| 3. **知识库** | 列 KB（含 0 条也算成功） | 未授权任何 KB（HMAC 模式默认看不到 public KB，需管理员显式授 `kb:public-read`） |

错误码细化（出现在右下角 toast）：

| 码 | 含义 | 用户操作 |
| --- | --- | --- |
| 4011 | 短期下载令牌过期 | 重新发起搜索，新的搜索结果自带新令牌 |
| 4014 | 凭据失效 / 用户被禁用 | 重新登录或联系管理员 |
| 4031 | 用户对该 KB 无读权限 | 联系 KB 持有者授予 reader / editor |
| 4032 | 应用对该 KB 无 grant | 联系管理员调 `/openapi/v1/admin/apps/{id}/kb_grants` 授权 |
| 4033 | grant 已过期 | 联系管理员重新授权或延长 expires_at |

### 2.3 灰度开关

如果远程知识库出问题、影响日常使用，可一键关停：

- 设置 → 知识库设置 → 顶部 banner 出现时点"立即启用"切换；或
- 浏览器 DevTools 控制台：

```js
import('/src/utils/featureFlags.js').then(m => m.setFlag('kbRemoteIntegration', false))
```

关停后：连接信息、绑定关系**不**丢失，仅在发送消息时跳过整个 KB 检索/引用环节。再次开启即恢复。

## 3. 在 WPS 文档里使用

### 3.1 选段触发（推荐）

选中正文一段或一句，三种入口效果一致：

- **Ribbon → 察元 AI → 知识库** → 选「核对本段 / 总结本段 / 问答本段」
- **右键菜单 → 察元AI知识库** → 同上 3 项（在表格单元格里也有同名菜单）

| 模式 | 效果 |
| --- | --- |
| `verify` 核对本段 | 模型对照 KB 内容判断"原文与知识库是否一致"，每条结论必须 `[^cN]` 引用对应片段 |
| `summarize` 总结本段 | 让模型用 KB 中相同主题的多个段落，对你选的段做"加 KB 上下文的总结" |
| `qa` 问答本段 | 把选段当作问题/任务，去 KB 检索可能的答案 |

> 如果未配置任何连接，会先弹连通性引导，再开知识库设置面板。

### 3.2 直接发送

打开察元 AI 助手 → 在输入框打你的问题 → 发送。只要对话已绑定 KB，即按 `qa` 模式跑检索。

## 4. 阅读结果

### 4.1 引用脚标 `[c1]`

模型回答里出现的每个 `[c1]`、`[c2]` 都是**可交互**的：

- **悬停**：底部"知识来源"折叠条里对应卡片高亮（双向）
- **点击**：折叠条自动展开并滚动到对应卡片

### 4.2 知识来源折叠条

每张卡片包含：

- **左上角**：`[c1]` 引用 ID + 5 档信任度星级（基于召回分、标题命中、章节命中、查询召回、来源质量、跨批次复现 6 个信号加权得到）
- **第二行**：KB 名 / 原始文件名（点击下载，30 分钟有效）
- **正文**：命中的关键片段（最多 280 字）
- **右下角**：tag 链 — 表示这条片段被哪些子查询命中（验证模式下还有所属章节标题）

### 4.3 长文档怎么办（自动）

选段超过 800 字时，前端自动用"四层漏斗"处理而不是一次性发：

| 层 | 何时启用 | 做什么 |
| --- | --- | --- |
| T1 | 总是 | 按 Markdown 标题 / 段落 / 句子结构切 |
| T2 | 总是 | 自适应归并相邻短片段 |
| T3 | > 4 单元 / > 800 字 | 用 ngram 向量做层次聚类，每簇取代表段 |
| T4 | > 20 000 字 | 调本地模型做关键短语蒸馏，每簇出 ≤ 3 个查询短语 |

T1+T2+T3 在 Web Worker 里跑，**不卡 UI**；T4 跑在主线程（需要 LLM 闭包）。

## 5. FAQ

**Q: 我同时绑了 3 个 KB，会怎么搜？**
A: 单次请求扁平化为"3 个 KB × N 个子查询"的笛卡尔积，服务端用并发信号量控制不冲爆，最后用 RRF 或 weighted 融合，dedup_by chunk_id。

**Q: 如果某个 KB 临时挂了？**
A: 部分失败容错 — 单 KB 失败只标 `partial`，其余正常融合。返回中会带 `degraded: true`。

**Q: 下载链接安全吗？会泄露吗？**
A: 链接里的 `?token=` 是一次性短期令牌（30 分钟过期），`aud` claim 严格匹配调用方主体（你的 user_id 或 app_id）。复制别人也用不了。

**Q: HMAC 模式签名怎么算？**
A: 见 `chayuan-server/docs/kb-search-batch-contract.md` §HMAC，前端 `services/kb/authClient.js` 已封装好，调用方不需要手算。

**Q: 我是管理员，要在哪里给应用授权 KB？**
A: WPS 加载项**不**做 admin UI（设计决定）。请走 chayuan-server 的 webui，或用 cURL：

```bash
curl -X POST 'https://kb.example.com/openapi/v1/admin/apps/<APP_ID>/kb_grants' \
  -H "Authorization: Bearer <ADMIN_JWT>" \
  -H "Content-Type: application/json" \
  -d '{"grants":[{"kb_name":"my-kb","role":"reader","expires_at":null}]}'
```
