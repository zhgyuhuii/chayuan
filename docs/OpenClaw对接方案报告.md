# niudang 项目与 OpenClaw 对接方案报告

## 一、报告目的

本文档用于整理 `niudang` 项目与 `OpenClaw / clawd-cursor` 的对接方案，明确接入目标、系统边界、所需配置、项目内落点、实施阶段与风险控制要求。

当前阶段目标为：

- 先完成方案设计与配置规划
- 不直接进入编码实现
- 以最小风险方式为后续接入预留结构

---

## 二、项目现状概述

根据当前项目结构判断，`niudang` 是一个基于 `Vue 3 + Vite + WPS JS API` 的 `WPS` 加载项项目，而非普通 Web 应用。

项目当前具备以下特点：

- 有明确的 `WPS` 插件属性
- 已存在 `Ribbon` 菜单入口
- 已存在设置页、任务窗格、任务编排页
- 已具备 `PluginStorage + localStorage + 文件路径` 的本地持久化设计
- 已有任务系统与工作流系统，可作为后续集成承载层

因此，`OpenClaw` 的接入不应设计成“替换现有 AI 能力”，而应设计成“为当前插件增加桌面自动化与跨应用执行能力”。

---

## 三、接入目标

### 3.1 核心目标

将 `OpenClaw / clawd-cursor` 作为本机桌面执行引擎接入 `niudang`，使插件在保留现有文档 AI 能力的基础上，新增以下能力：

- 打开外部应用
- 打开浏览器并访问指定页面
- 在系统层执行桌面自动化动作
- 为任务编排提供跨应用节点能力
- 为 WPS 插件提供本地桌面级操作扩展

### 3.2 非目标

以下内容不建议在第一阶段纳入：

- 不替代现有 `chatApi`、模型调用链或媒体生成链
- 不把 `OpenClaw` 作为项目主 AI 编排核心
- 不一开始做双向复杂事件回调
- 不一开始开放完全自由的高风险自然语言执行
- 不默认允许邮件、删除、消息发送等敏感动作

---

## 四、推荐接入架构

推荐采用如下结构：

```text
niudang（WPS 插件前端）
  -> 本机 HTTP 调用
clawd-cursor 本地服务
  -> 作为 OpenClaw skill 或独立桌面自动化服务运行
OpenClaw / 桌面自动化能力 / 已配置模型提供方
```

### 4.1 架构说明

在该方案中：

- `niudang` 负责任务发起、配置管理、结果展示、审批入口和任务编排
- `clawd-cursor` 负责本机桌面执行与 OpenClaw 技能能力承接
- `OpenClaw` 负责 skill 生态、provider 发现与桌面代理能力扩展

### 4.2 为什么采用本机桥接模式

此方案适合当前项目的主要原因如下：

- `niudang` 属于 `WPS` 插件场景，运行环境受宿主限制
- 当前项目已具备本地存储与任务体系，适合接入“外部本机服务”
- `clawd-cursor` 已提供本机 HTTP 接口，接入边界清晰
- 可以避免把 OpenClaw 内部运行逻辑直接耦合进前端插件
- 安全策略可以在项目内单独收口

---

## 五、系统职责划分

### 5.1 niudang 负责

- 发起桌面任务
- 展示任务执行状态
- 保存集成配置
- 提供用户确认入口
- 在 `Ribbon`、设置页、任务编排页中暴露能力
- 对高风险任务做前置拦截与白名单控制

### 5.2 clawd-cursor 负责

- 接收自然语言或结构化任务
- 调用本机桌面自动化能力
- 操作浏览器、桌面应用和系统界面
- 返回任务执行状态与结果
- 承接来自 `OpenClaw` 的 skill 运行模式

### 5.3 OpenClaw 负责

- skill 安装与统一管理
- 已配置 AI provider 的自动发现
- 为 `clawd-cursor` 提供技能运行上下文
- 作为更高层的 AI agent 平台存在

---

## 六、建议的能力边界

为了避免能力重叠和架构混乱，建议明确边界如下：

### 6.1 继续保留在 niudang 内部的能力

- 文档摘要
- 文本翻译
- 拼写与语法检查
- 文本分析
- 表单编辑
- 文档审查
- 文档内工作流编排
- AI 模型设置与供应商管理

### 6.2 交给 OpenClaw / clawd-cursor 的能力

- 打开 Cursor 并打开指定项目
- 打开浏览器并访问外部系统
- 跨应用复制、粘贴、导航
- 启动桌面应用
- 执行系统级 UI 操作
- 执行需要桌面可见环境的外部操作

### 6.3 后续可组合的能力

- 从 `niudang` 发起桌面动作后再回到文档处理
- 在工作流中插入“桌面任务节点”
- 打开外部知识库页面后抓取结果并回写文档

## 七、运行环境配置方案

### 7.1 OpenClaw skill 模式

如果环境中已经使用 `OpenClaw`，建议优先采用 skill 模式：

```bash
openclaw skills install clawd-cursor
```

此模式的优点：

- `clawd-cursor` 能自动发现 OpenClaw 中已配置的 provider
- 不需要在 `niudang` 中重复维护桌面自动化所需模型配置
- 后续技能扩展和能力升级更自然

### 7.2 独立服务模式

如果暂时不通过 OpenClaw skill 统一管理，也可以先独立启动本机服务：

```bash
git clone https://github.com/AmrDab/clawd-cursor.git
cd clawd-cursor && npm install && npm run setup
clawdcursor doctor
clawdcursor start --port 3847
curl http://127.0.0.1:3847/health
```

### 7.3 macOS 额外配置

当前使用环境为 `macOS`，因此必须完成以下系统授权：

- 打开“系统设置”
- 进入“隐私与安全性”
- 进入“辅助功能”
- 为 `Terminal` 或实际启动 `clawd-cursor` 的终端工具授权

如后续改为其他启动方式，也需要为对应应用补充授权。

### 7.4 推荐固定运行参数

建议统一按如下方式约定：

- 服务地址：`http://127.0.0.1:3847`
- 健康检查接口：`/health`
- 任务提交接口：`/task`
- 审批接口：`/confirm`
- 中止接口：`/abort`
- 默认安全级别：`confirm`

## 八、项目侧配置设计

建议在项目中单独增加“OpenClaw 集成设置”，不要与现有模型供应商设置混放。

### 8.1 建议配置项

建议至少包含以下字段：

| 配置项 | 类型 | 建议值 | 说明 |
|--------|------|--------|------|
| `enabled` | boolean | `false` | 是否启用 OpenClaw 集成 |
| `baseUrl` | string | `http://127.0.0.1:3847` | 本机服务地址 |
| `healthPath` | string | `/health` | 健康检查接口 |
| `taskPath` | string | `/task` | 任务提交接口 |
| `confirmPath` | string | `/confirm` | 审批接口 |
| `abortPath` | string | `/abort` | 中止接口 |
| `timeoutMs` | number | `10000` 或 `15000` | 请求超时 |
| `transportMode` | string | `local-http` | 传输方式 |
| `safetyMode` | string | `confirm` | 安全等级 |
| `requireUserApproval` | boolean | `true` | 高风险操作需确认 |
| `allowedDomains` | string[] | 按需配置 | 允许访问的网站白名单 |
| `allowedApps` | string[] | 按需配置 | 允许操作的应用白名单 |
| `blockedTaskKeywords` | string[] | 按需配置 | 高风险关键字拦截 |
| `enableInWorkflow` | boolean | `false` | 是否允许在任务编排中使用 |
| `enableInRibbon` | boolean | `true` | 是否显示在 Ribbon 中 |
| `logTaskHistory` | boolean | `true` | 是否记录执行历史 |
| `storeTaskPayload` | boolean | `false` | 是否保存原始任务文本 |

### 8.2 存储建议

建议采用以下存储策略：

- 首选 `PluginStorage`
- 其次同步 `localStorage`
- 必要时再落本地配置文件

原因：

- `PluginStorage` 更符合当前 `WPS` 插件运行场景
- 当前项目已经存在此类存储设计
- 便于在 `Ribbon`、任务窗格和设置页之间共享配置

## 九、项目内落点设计

### 9.1 设置页

建议在设置页中新增一个独立区域：

- 名称建议：`OpenClaw 集成`
- 用途：配置地址、测试连接、查看状态、设置安全策略、维护白名单

建议包含以下 UI：

- 启用开关
- 服务地址输入框
- 健康检查按钮
- 连接状态展示
- 安全级别选择
- 白名单域名配置
- 白名单应用配置
- 高风险词拦截配置

### 9.2 Ribbon 入口

建议在顶部 `Ribbon` 中新增一个轻量入口，例如：

- `桌面任务`
- `OpenClaw 执行`
- `打开外部工具`

适合作为快速触发入口，但不适合承载复杂配置逻辑。

### 9.3 任务系统

建议将 OpenClaw 执行结果纳入现有任务系统，使其与现有 AI 任务统一管理：

- 运行中
- 已完成
- 已失败
- 已取消
- 待确认

这样可以最大化复用当前任务清单、任务状态、结果查看与错误展示能力。

### 9.4 任务编排页

建议在第二阶段后，将 OpenClaw 设计为一种新的工作流节点类型。

适合的节点类型包括：

- 打开网页
- 打开指定应用
- 执行桌面流程
- 打开 Cursor 并定位项目
- 打开浏览器后跳转到指定知识库

### 9.5 工具层

建议在 `src/utils` 中新增独立服务模块，例如：

- `openclawClient`
- 或 `desktopAutomationService`

不建议直接放进现有：

- `chatApi`
- `mediaApi`
- `assistantTaskRunner`

原因是职责边界不同，后续维护也更清晰。

## 十、第一阶段建议落地的能力

为控制复杂度，建议第一阶段仅实现最小闭环。

### 10.1 连接测试

在设置页中支持：

- 测试 `baseUrl` 是否可访问
- 查询 `/health`
- 展示版本、在线状态、端口信息

### 10.2 任务发送

提供一个最简单的任务发起入口，例如：

- 打开浏览器并访问某个地址
- 打开 Cursor 并打开当前项目

### 10.3 状态展示

接入任务系统，展示：

- 发起时间
- 执行状态
- 返回结果
- 错误信息

### 10.4 安全拦截

第一阶段必须具备：

- 白名单校验
- 高风险关键字拦截
- 用户确认机制

## 十一、第二阶段建议能力

- 支持模板化桌面任务

- 支持“执行完成后回写文档”

## 十二、典型场景建议

### 场景一：打开浏览器访问指定页面

适用场景：

- 打开企业门户
- 打开知识库
- 打开外部模型文档
- 打开项目仓库

### 场景二：打开 Cursor 并打开项目

适用场景：

- 从 WPS 插件中直接跳转到当前工程开发环境

### 场景三：工作流中的桌面节点

适用场景：

- 文档分析后打开外部页面补充资料

### 场景四：任务清单统一管理

适用场景：

- 用户在插件内统一查看 AI 任务与桌面任务
- 统一失败重试与错误追踪

## 十三、安全与风险控制

由于 `OpenClaw / clawd-cursor` 涉及桌面自动化，因此安全策略必须前置。

### 13.1 默认原则

- 默认不开启完全自动执行
- 默认只允许本机 `localhost`
- 默认启用用户确认
- 默认不保存敏感任务原文

### 13.2 建议拦截的高风险动作

- 删除文件
- 覆盖文件
- 发送消息
- 发送邮件
- 转账付款
- 打开敏感账号页面
- 操作聊天工具联系人

### 13.3 白名单策略

建议建立两类白名单：

- 域名白名单
- 应用白名单

只有在白名单中的站点和应用才允许自动触发。

### 13.4 日志策略

建议默认只保存：

- 任务标题
- 状态
- 时间
- 简要结果

不建议默认保存：

- 完整原始任务
- 敏感输入内容
- 桌面截图

除非后续有明确审计场景再做增强。

---

## 十四、不建议的设计方式

以下设计不建议采用：

### 14.1 让 OpenClaw 替代当前文档 AI 主链路

原因：

- 当前项目已有成熟的 AI 功能与模型设置体系
- 混用后职责边界会变得模糊

### 14.2 一开始就支持任意自然语言全量执行

原因：

- 风险过高
- 容易出现越权动作
- 难以控制测试边界

### 14.3 一开始就实现双向事件回调和复杂编排

原因：

- 实现复杂度高
- 在 `WPS` 插件环境中调试成本更高
- 第一阶段价值不如“先打通最小闭环”

## 十五、建议的实施阶段

### 第一阶段：环境打通

- 安装 `clawd-cursor`
- 配置 `OpenClaw skill` 或独立本机服务
- 完成 `macOS` 辅助功能授权
- 确认 `/health` 正常

### 第二阶段：项目配置接入

- 增加 OpenClaw 集成配置结构
- 接入设置页
- 实现连接测试

### 第三阶段：最小任务闭环

- 在插件内发起简单桌面任务
- 接入任务系统
- 展示执行状态和结果

### 第四阶段：入口扩展

- 增加 `Ribbon` 快捷入口
- 增加模板化任务入口

### 第五阶段：工作流集成

- 在任务编排中增加 OpenClaw 节点
- 支持跨应用流程组合

### 第六阶段：安全增强

- 完善确认流
- 完善白名单与风险词策略
- 增加日志脱敏与审计控制

---

## 十六、最终建议

综合当前项目结构与 OpenClaw 的能力形态，推荐采用以下明确方案：

- 接入模式：`niudang -> localhost clawd-cursor REST API`
- OpenClaw 角色：provider 与 skill 运行环境上游
- 项目角色：发起任务、保存配置、展示结果、实施安全控制
- 存储策略：`PluginStorage` 为主，`localStorage` 与本地文件为辅
- UI 落点：`设置页 + Ribbon + 任务编排页`
- 默认安全策略：`confirm`
- 第一阶段目标：`health check + send task + task list`

这是当前项目最稳妥、最清晰、后续扩展空间也最大的方案。
