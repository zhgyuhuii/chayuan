### 当前真实剩余偏差

- `普通聊天`：动态预算策略（light/standard/tight 三档）、长期记忆（chatMemoryStore）、摘要质量抽检标记已全部落地。差距：摘要仍为规则型压缩，还没有独立的人工抽检/复核面板。
- `文档处理`：逐操作级账本（documentOperationLedger）和设置页逐操作预览/重放/导出 UI 已落地，大文档质量门禁（buildQualityGate）也已接入。差距：重放动作覆盖还不完全（格式类操作、删除操作尚不支持重放）。
- `原生能力 / 扩展能力`：执行前门禁（capabilityPolicyStore）已在 capabilityBus 和 wpsCapabilityExecutor 中生效，风险分级、高风险确认拦截、审计聚合与导出均已落地。差距：设置页还没有策略配置 UI，用户无法可视化管理策略。
- `文件生成 / 多模态识别`：已拆出 `audio-understanding` 模型类型，音频走 ASR 转写 + 理解摘要双层流水线，视频走动态采样 + 分段 + 模型元数据。差距：`useServerFallback` 只打了标志，没有真正的服务端抽帧调用链。
- `助手修复 / 进化`：健康分（buildAssistantHealthScore）和发布门禁（buildAssistantReleaseGate）已嵌入创建、修复、进化、晋升全路径，并在评测记录和设置页中可查。差距：还不是"大规模自动双跑 + 回归工作台"的完整评测平台。

你现在已经有报告模板和预设，这是优势，建议继续强化“先结构化、再渲染”。

最佳实践是：

模型先输出结构化 report.json
再用 renderer 生成：
report.md
report.docx
report.html
report.json
不要让模型直接一次性产出最终 Word 风格长文，否则可控性差。

7. Excel / Markdown / JSON 文件生成建议
这是当前项目非常适合补齐的一块。

Excel
不要让模型直接“生成 xlsx 二进制”，而是先生成工作簿结构：

{
  "sheets": [
    {
      "name": "问题清单",
      "columns": ["序号", "问题", "风险级别", "建议"],
      "rows": [
        [1, "合同主体缺失", "高", "补全主体名称"]
      ]
    }
  ]
}
再由本地 renderer 用 xlsx 落成真实文件。
你项目里已经有 xlsx 依赖，这条很适合快速补上。

Markdown
最容易落地，建议统一支持：

摘要
审计报告
会议纪要
清单类结果
JSON
JSON 应该成为所有复杂产物的“中间协议”，不是只当调试输出。
